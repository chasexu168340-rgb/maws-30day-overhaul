# MAWS 场景 UI 视觉规范

## 目标

场景 UI 的目标是让玩家先看见当前场景、当前角色和当前选择，再看见解释、历史和收益细目。它服务于《了不起的武术模拟器》的核心循环：进入场景、识别局势、点击对象、做出选择、承受结果、推进下一步。

本规范只定义视觉呈现和信息层级，不新增系统宽度，不改变战斗、经济、事件、存档或资产结构。

## 场景舞台

### 角色比例

- 主角/NPC 使用同一套场景比例基准，避免 UI 化头像和舞台角色比例混用。
- 主角在常规对话/事件场景中应略强于普通 NPC：主角视觉高度约为普通 NPC 的 105%-115%。
- 关键 NPC、Boss、主线压迫角色可放大到普通 NPC 的 120%-140%，但必须服务于剧情压迫或选择压力，不能作为常规排版手段。
- 普通交互 NPC 不应超过主角尺寸；弱小、路人或远景 NPC 可缩小到主角的 75%-95%。
- 同屏角色数量以可读为先。桌面端通常 1 个主角 + 1-3 个 NPC；移动端通常 1 个主角 + 1 个当前交互对象，其他角色折叠到列表或背景提示。

### 脚底接触阴影

- 所有站立角色必须有脚底接触阴影，避免角色漂浮。
- 接触阴影应贴近脚底，椭圆形或压扁形，透明度低于角色主体。
- 阴影大小跟随角色尺度变化：主角和重要 NPC 阴影更清晰，远景/弱化 NPC 阴影更浅。
- 阴影只表达接地关系，不承担大面积背板或信息分区功能。

### 轮廓光

- 主角、当前说话人、当前可点击对象可使用轻量轮廓光。
- 轮廓光优先用于聚焦：当前说话人最强，主角次之，可点击对象以 hover/selected 状态增强。
- 红色轮廓光只用于危险、主线压迫、敌意、代价或失败风险。
- 金色轮廓光只用于关键奖励、主线强调、稀有机会或重要推进。
- 灰色/白色轮廓光只用于普通可交互状态和焦点可读性。
- 禁止所有角色常亮同强度轮廓光；没有层级就等于没有引导。

### 背景与背板

- 场景舞台必须保留可玩的空间感，不能被通用仪表盘式卡片布局覆盖。
- 禁止使用大面积红、黄、灰背板作为默认场景背景。
- 红色只能做危险和主线压迫的语义强调，不能铺满普通事件。
- 金色只能做关键收益和主线重点的语义强调，不能铺满普通奖励。
- 灰色只能做次级信息、历史记录、说明文本或失效状态，不能成为主要舞台气氛。
- 信息容器应靠边、半透明、局部遮罩或折叠处理，不能把角色和可点击对象压成背景装饰。

## 对话框

### 当前台词优先

- 对话框第一优先级是当前台词，不是背景设定、历史记录或收益结算。
- 当前台词应位于对话框主阅读区，字号、对比度和留白高于历史文本。
- 当前台词过长时优先分页或折叠后文，不允许压缩到难读的小字。
- 情绪、威胁、邀请、失败提示等当前节拍必须直接可见。

### 说话人清楚

- 说话人姓名必须与当前台词绑定显示，不能只依赖角色站位或头像推断。
- 当前说话人应在舞台上同步高亮，可用轻量轮廓光、呼吸动效、朝向或名字牌强化。
- 多人对话必须避免多个名字牌竞争。非当前说话人降级为背景角色、头像列或折叠名单。
- 旁白、系统提示、战报式文本必须与角色台词在视觉上区分。

### 历史和收益折叠

- 对话历史默认折叠，只暴露入口和最近一条必要上下文。
- 收益、代价、检定细节默认折叠到次级区域，但关键变化必须即时可见。
- 展开历史或收益后不能挤压当前台词到不可读；应使用抽屉、浮层或切换标签。
- 长解释、世界观补充、完整结算明细都属于折叠内容，不应占据当前选择的主位置。

## 事件反馈

### 点击后必须有结果反馈

- 玩家点击场景事件、角色或选项后，必须立刻看到结果反馈。
- 反馈至少包含一个明确变化：台词改变、状态标签变化、收益/代价数字变化、角色反应、场景特效或下一步按钮变化。
- 不能让点击只关闭弹窗、只刷新列表、只写入日志而没有当前反馈。
- 失败也必须有内容：失败台词、后果说明、替代路线、惩罚或新限制必须可见。

### 收益/代价即时可见

- 收益和代价必须在结果发生后即时显示，不依赖玩家打开历史日志查找。
- 关键收益使用金色强调，关键代价/危险使用红色强调，普通变化使用中性文字或小型标签。
- 数值变化应显示方向和对象，例如 `声望 +2`、`体力 -8`、`获得：残页`。
- 如果结果包含多项变化，主收益/主代价露出，完整明细折叠。

### 后续选择明确

- 结果反馈后，下一步选择必须清楚：继续、离开、追问、战斗、支付、放弃、返回等命令要有明确文案。
- 禁止只给抽象按钮，例如只有“确定”却无法判断下一步会发生什么。
- 高风险选择要提前暴露风险语义，不能只在点击后才显示代价。
- 已不可选的后续选项应说明原因，使用灰色次级状态，不与可点击按钮混淆。

## P5 / BG3 / DOS2 取舍

### 采用 P5 的部分

- 采用强视觉层级：当前台词、当前说话人、当前选择、主线压力必须一眼分明。
- 采用主线压迫感：红色、倾斜构图、强对比和节奏动效可用于危险、截止、敌意、重大抉择。
- 不采用纯装饰化高饱和铺屏。P5 式冲击只用于推动局势，不用于普通信息面板。

### 采用 BG3 的部分

- 采用选择后果清楚的呈现：选项前后都要让玩家理解风险、收益和关系变化。
- 采用失败也有内容：失败不是空提示，而是新的台词、代价、分支、局势恶化或替代路径。
- 不照搬复杂骰检面板。MAWS 只保留必要的风险和结果表达，避免新增系统负担。

### 采用 DOS2 的部分

- 采用场景点击和角色交互：可点击对象应在场景中被看见、被指向、被反馈。
- 采用角色和环境共同讲述事件：结果可以通过角色动作、站位、光效、标签和对话变化共同呈现。
- 不采用把所有交互压进列表的做法。列表可以辅助，但不能替代场景对象和角色关系。

## 移动端

### 不挤多个角色

- 移动端优先保留当前交互角色和主角，其他 NPC 降级为头像列、背景剪影、名单或折叠入口。
- 禁止为了保留桌面端阵容而把多个角色缩到不可读。
- 横屏和竖屏都要保护脚底、轮廓、名字牌和点击热区，不能互相覆盖。

### 弹窗优先当前内容

- 移动端弹窗只优先当前台词、当前结果、当前选择。
- 历史、完整收益、世界观解释、长战报放入折叠区。
- 弹窗高度不足时优先滚动次级内容，不让当前台词和主按钮被挤出屏幕。
- 底部导航和核心退出路径必须保持可见或有明确返回入口。

### 长文必须折叠

- 移动端长文默认折叠，首屏只展示当前节拍和必要决策信息。
- 单段文本过长时拆成短段、分页或“展开详情”。
- 展开长文后必须能回到当前选择，不允许玩家读完后丢失操作上下文。
- 历史、收益明细、失败原因、世界观补充都应有明确折叠标题。

## 验收清单

- 主角/NPC 比例有层级，移动端不硬塞多人。
- 所有站立角色有脚底接触阴影。
- 当前说话人、主角、可点击对象有可区分的轮廓光规则。
- 没有大面积红黄灰背板作为默认舞台。
- 对话框优先当前台词，说话人清楚，历史和收益折叠。
- 点击事件后有结果反馈，收益/代价即时可见，后续选择明确。
- P5、BG3、DOS2 只被用于强化 MAWS 核心循环，不引入无关系统宽度。
