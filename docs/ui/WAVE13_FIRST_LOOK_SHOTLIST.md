# Wave 13 First-Look Shotlist

适用对象：人工验收、视觉 QA、后续 UI/美术实现回归。
验收目标：第一眼能判断这是一段可玩的武术模拟切片；主要反馈清楚、点击有回应、战斗 HUD 不遮挡核心画面。

## 全局拍摄要求

- 桌面：优先使用 1440x900 或 1366x768，截图包含完整游戏窗口、底部导航和当前主内容。
- 手机：优先使用 390x844 或 375x812，截图包含顶部状态、主内容和底部导航。
- 每张截图保留真实 UI 状态，不裁掉边缘，不只截局部组件。
- 失败截图需要保留问题发生时的完整上下文，例如遮挡、溢出、反馈丢失或按钮不可点的位置。

## 1. 出租屋主界面

### 截图角度

- 桌面：进入游戏后停留在出租屋/本地点主界面，主角、地点背景、可执行行动、资源状态和底部导航都入镜。
- 手机：竖屏进入同一界面，确认主行动区和底部导航同时可见。

### 通过标准

- 第一眼能看出当前地点是出租屋，主视觉不是空白面板或纯调试信息。
- 主角/角色视觉在画面中有明确落点，不显得漂浮、过小或被 UI 挤到边缘。
- 当前可做的行动在 3 秒内可被识别，主按钮或行动入口有清楚层级。
- 资源、日期、体力/状态等关键信息可读，不和背景或装饰混在一起。
- 底部导航可见且不遮挡本地点行动按钮。

### 不通过标准

- 画面主要由密集文字表格组成，缺少可识别的地点或角色主视觉。
- 主行动按钮被底栏、弹层或滚动容器遮住。
- 桌面端出现横向溢出，或手机端需要横向滚动才能看到核心按钮。
- 角色像贴纸悬浮在背景上，比例明显小于 UI 卡片重点。
- 重要资源文字低对比、重叠、截断或被背景纹理吞掉。

### 手机/桌面注意点

- 手机端允许信息分组纵向堆叠，但行动入口不能掉到首屏之外太深。
- 桌面端不能把主界面拉成稀疏空白，也不能让侧栏/卡片抢走角色和地点焦点。
- 两端都要检查底部导航点击区至少保持可用，不被滚动内容覆盖。

## 2. 点击 NPC/角色

### 截图角度

- 桌面：点击 NPC 或主角后的瞬间，保留角色、点击反馈、相关信息面板或浮层。
- 手机：用触屏尺寸截图点击反馈后的状态，保留点击目标和反馈位置。

### 通过标准

- 点击后有明确视觉反馈，例如高亮、轻微缩放、描边、选中态、角色信息面板或上下文行动入口。
- 反馈和被点击角色之间关系明确，用户能判断“刚才点中了谁”。
- 如果出现文本反馈，内容短而具体，不依赖长 toast 才能理解结果。
- 反馈不遮挡关键行动按钮，不抢走底部导航可点击性。
- 连续点击不同角色时，选中态能正确转移，不残留多个冲突状态。

### 不通过标准

- 点击后只有短暂 toast，角色本身没有任何选中或响应状态。
- 反馈位置远离角色，无法判断和哪个 NPC 有关。
- 选中面板覆盖角色大半身位，或让主行动入口不可点。
- 手机端点击目标太小，误触底部导航或相邻按钮。
- 桌面端 hover 有效果但实际 click 没有稳定状态。

### 手机/桌面注意点

- 手机端优先检查触控命中范围和反馈持续时间，反馈不能只依赖 hover。
- 桌面端同时检查 hover、active、selected 三类状态是否彼此一致。
- 小屏下角色信息可以压缩，但不能把可执行操作藏进不可见区域。

## 3. 普通事件收益反馈

### 截图角度

- 桌面：完成一次普通事件或行动后，截取收益反馈出现的完整状态。
- 手机：同一类事件完成后，截取反馈、资源栏和可继续操作入口。

### 通过标准

- 收益以结构化短条目呈现，能快速读出金钱、物品、关系、技能、风险或时间变化。
- 数字变化有明确正负、单位或语义，例如 `+12 铜钱`、`-1 时辰`、`关系 +2`。
- 反馈不重复长句，不把关键收益藏在段落中。
- 反馈出现后，玩家能明确下一步可做什么，例如关闭、继续、返回行动。
- 资源栏或相关状态与收益反馈不矛盾。

### 不通过标准

- 仍以长句黄块或整段叙事承担主要收益说明。
- 多个收益混成一行，无法分辨哪个是成本、哪个是奖励。
- 数字缺少符号或标签，玩家不能判断是增加还是扣除。
- 反馈弹层过大，普通事件看起来像主线大事件。
- 手机端收益 chip 换行后重叠、截断或挤压关闭按钮。

### 手机/桌面注意点

- 手机端收益条目要允许换行或纵向排列，不能产生横向滚动。
- 桌面端反馈可以更宽，但仍要保持短标签和可扫读的数字层级。
- 两端都要检查反馈出现后是否还能继续推进状态。

## 4. 时间投入弹窗

### 截图角度

- 桌面：打开需要投入时间的行动弹窗，保留标题、说明、时间成本、确认/取消按钮和背景上下文。
- 手机：竖屏打开同一类弹窗，确认底部按钮完整可见。

### 通过标准

- 弹窗标题直接说明将要做的行动，时间成本在确认前可见。
- 成本和收益预期分区清楚，不需要读完整段说明才能知道会花多少时间。
- 确认、取消按钮层级明确，危险或高成本操作有足够提示。
- 背景被弱化但仍能看出来自哪个地点或行动。
- 键盘/触控焦点不会落到弹窗外的底部导航。

### 不通过标准

- 时间成本只出现在确认后的结果里，确认前不可见。
- 弹窗高度超过手机屏幕且主要按钮被底部遮挡。
- 普通时间投入弹窗使用过大的剧情弹窗样式，影响操作节奏。
- 关闭、取消、确认按钮位置混乱，容易误点。
- 桌面端弹窗宽度过大，正文行长难读。

### 手机/桌面注意点

- 手机端确认按钮必须在安全区内，滚动到底也不能被浏览器或游戏底栏盖住。
- 桌面端弹窗应居中或贴近触发区域，避免离当前行动太远。
- 两端都要检查取消后状态不推进，确认后时间和收益反馈同步出现。

## 5. Day 5 战斗 HUD

### 截图角度

- 桌面：进入 Day 5 战斗，截取完整战斗舞台、双方角色、血量/状态、行动卡或技能区。
- 手机：竖屏进入同一战斗，截取角色、HUD、行动入口和底部安全区。

### 通过标准

- 战斗舞台是画面焦点，双方角色和背景关系清楚，角色不被 HUD 吃掉。
- 血量、状态、回合/行动提示可读，和角色位置关系明确。
- 卡牌或技能区不占据半个屏幕以上，不遮挡主要打斗区域。
- 当前可执行动作有明显可点状态；不可用动作有明确 disabled 状态。
- 战斗反馈能表达命中、受伤、格挡或结算变化，不只改数字。

### 不通过标准

- 卡牌/HUD 占据画面主要面积，角色只能看到很小或被遮住。
- 血条、状态文本和背景低对比，无法快速判断战况。
- 手机端行动区挤压战斗舞台，角色或敌人掉出首屏。
- 桌面端角色仍像贴纸或漂浮，没有贴地和舞台关系。
- 关键操作按钮低于底部安全区，或被底部导航覆盖。

### 手机/桌面注意点

- 手机端优先保留战斗舞台和一个明确主行动区，次要信息可以折叠或缩短。
- 桌面端要利用宽度展示双方关系，但不能把 HUD 横向铺满到喧宾夺主。
- 两端都要检查战斗开始、行动选择、受击反馈、结算前后的 HUD 稳定性。

## 验收记录建议

- 每个画面至少提交 1 张桌面截图和 1 张手机截图。
- 每张失败截图配一句原因，格式建议：`画面编号 - 设备 - 失败标准 - 简短说明`。
- 若同一问题跨多个画面出现，归为系统性问题；若只在单一尺寸出现，标注具体 viewport。
